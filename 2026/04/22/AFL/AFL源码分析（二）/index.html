这一篇我们来看AFL最核心的源码：afl-fuzz.c，我们还是按照main函数的顺序来看各个函数，最后贴出main函数代码。、

setup_signal_handlers

该函数功能较为简单

  1. 将SIGHUP、SIGINT、SIGTERM信号的处理函数注册为handle_stop_sig
  2. 将SIGALRM信号处理函数注册为handle_timeout
  3. 将SIGWINCH信号处理函数注册为handle_resize
  4. 将SIGUSR1信号处理函数注册为handle_skipreq
  5. 将SIGTSTP、SIGPIPE信号处理函数注册为SIG_IGN

chack_asan_opts

功能如其函数名：检查ASAN选项：

  1. 检查环境变量ASAN_OPTIONS的值，检查其中是否包含abort_on_error=1或symbolize=0：其中，abort_on_error=1的含义：当 ASan 检测到内存错误（如 heap-buffer-overflow、use-after-free 等）时，立即调用 abort() 终止进程，而不是仅打印错误信息后继续执行（默认行为）。AFL依赖子进程的非零退出状态或信号终止（如 SIGABRT）来判断是否触发了 bug。symbolize=0的含义：禁用运行时符号化解析。ASan/MSan 在报告错误时通常会尝试将地址转换为函数名、文件名、行号。AFL需要将其设置为0，这样可以显著提高fuzzing的性能。
  2. 检查环境变量MSAN_OPTIONS的值，检查其中是否包含exit_code=或symbolize=0：其中exit_code=这个是当MSan检测到未初始化内存读取时，以多少的退出码退出进程，AFL要求必须设置为固定值，以检测是否出错。symbolize=0的含义：当ASan/MSan检测到内存错误（溢出，未初始化等）时，默认会尝试将错误栈中的虚拟地址转化为函数名，源码名和行号，这个过程会降低Fuzzing的效率，所以AFL要求设置为0。

fix_up_sync

当使用-S选项的时候，验证并修复out_dir和sync_dir。
在这个函数之前，main函数会解析命令行参数，解析到-S选项的时候，就会设置sync_id（被称之为从属模块的同步ID），我们来看一下该函数做了什么操作：

  1. 检查dumb_mode标志位是否被设置（当使用-n选项的时候，会设置该标志位），如果设置了，提示：-S / -M 和 -n选项是互斥的。
  2. 先检查skip_deterministic标志位，当使用-d选项时，该标志位会被设置，代表跳过确定性变异阶段，然后检查force_deterministic标志位，当使用-M选项时，该标志位会被设置(==为什么该标志位被设置？因为主节点需执行完整变异以生成高质量测试用例供从节点使用==）如果两个都被设置，提示使用-S选项替代-M -d选项，如果只设置了skip_deterministic选项，则提示-S已经包含了-d。
  3. 校验sync_id的合法性，允许下划线，连字符，数字，字母，而且长度不能超过32：
  1. 如果通过校验，则设置sync_dir = out_dir,out_dir = sync_id
  2. 如果没有使用-M选项，则帮助开启优化：

fix_up_banner

该函数用于更新banner，以便在控制台显示。

  1. 如果没有指定-T选项，则检查是否开启-s或者-M（检查sync_id是否被设置），如果被设置，则将use_banner设置为sync_id，如果没有设置sync_id标识，则去掉路径前缀。
  2. 如果use_banner长度超过40，则截断前40字节，拼接...后保存。

check_if_tty

该函数检测是否需要运行UI界面

  1. 检查环境变量AFL_NO_UI，如果该环境变量被设置，则标识不需要UI
  2. 检查stdout是否在tty上，如果没有的话，设置no_on_tty = 1

get_core_count

该函数用来获取系统CPU核心数，并判断当前系统负载，我们以Linux系统为例：

  1. 打开/proc/stat文件，逐行读取，统计以cpu开头且第一个字符为数字的行数，cpu_core_count++，这里的逻辑是：文件中比如cpu0,cpu1…都计为一个CPU核心数
  2. 成功获取CPU核心数之后，调用get_runnable_processes函数，获取当前运行任务数
  3. 计算当前负载如果cur_runnable > cpu_core_count * 1.5*，提示可能不稳定
    如果处于低负载，提示可启动并行fuzz实例

check_crash_handling

该函数功能为：确保核心转储不会被转发到程序。
AFL 依赖 waitpid() 系统调用来获取子进程退出状态：

  • 若程序 crash（如 SIGSEGV），内核会立即终止进程并返回状态。
  • 但如果系统启用了用户态崩溃处理器（如 macOS Crash Reporter 或 Linux 的 core_pattern 指向管道），内核会先 fork 并 exec 外部处理程序来生成 crash report，延迟子进程的真正终止。
  • 这导致 AFL 在 waitpid() 上等待超时（默认几秒），从而将 crash 误判为 hang。
    判断逻辑（以Linux为例）：
  1. 打开 /proc/sys/kernel/core_pattern。
  2. 读取第一个字符：若为 '|'，表示 core dump 被重定向到用户态程序（如 |/usr/share/apport/apport %p ...）。

check_cpu_governor

该函数功能就是检查CPU频率调解器。

  1. 检查环境变量AFL_SKIP_CPUFREQ，如果设置了该环境变量，则直接return
  2. 打开/sys/devices/system/cpu/cpu0/cpufreq/scaling_governor，如果打开失败，直接return
  3. 读取128字符到tmp中，如果前四个字符为perf，则代表目前处于高性能状态，直接return
  4. 打开/sys/devices/system/cpu/cpu0/cpufreq/scaling_min_freq，读取存入min
  5. 打开/sys/devices/system/cpu/cpu0/cpufreq/scaling_max_freq，读取存入max
  6. 如果min == max，说明频率被锁定，return
  7. 否则，提示系统启用了动态调频（如 ondemand），CPU 频率会在 [min, max] 之间调整。

setup_post

给 AFL 动态加载一个“后处理器（postprocessor）插件”，用来在每次测试用例送进目标程序之前，对输入做二次加工。
这个函数做的事情可以总结为一句话：

如果用户通过环境变量指定了一个 so 库，就把它动态加载进来，找到 afl_postprocess 这个函数，当作“输入变形器”使用，并提前跑一遍做健全性检查。

这是 AFL 给高级玩家准备的“外挂接口”。
我们来详细看一下该函数：

  1. 获取环境变量AFL_POST_LIBRARY，如果没有的话，直接return
  2. 加载动态链接库，立即解析所有符号，并查找约定好的函数afl_postprocess
  3. 如果加载成功，先调用一次，并输出成功信息（这里实际上是提前试错）

setup_shm

该函数虽然简单，但是支撑了AFL-fuzz整个底层，它解决的核心能力不是“变异”，而是，我刚才的输入，让程序走了那些路径？我们来详细看一下：

也就是说，这段共享内存，就是AFL感知路径反馈的神经网络。

setup_dirs_fds

该函数负责创建 AFL 运行所需的全部目录结构、做并发互斥、防误 resume、并打开若干关键文件描述符。

  1. 如果标志位sync_id被设置，那么创建sync_dir
  2. 创建out_dir，这是输出目录
    如果创建失败，但是文件存在，那么调用maybe_delete_out_dir函数（该函数在后面分析）
    如果创建成功，但是设置了in_place_resume标志，报错退出，如果没有设置，尝试对文件夹进行加锁
  3. 在out_dir目录下创建queue目录
    在queue目录下，创建：
    1. .state：用于会话恢复和相关任务的队列元数据顶级目录
    2. .state/deterministic_done：用于标记过去，已经经过确定性模糊测试的队列条目的目录
    3. .state/auto_extras：包含自动选择的词典条目的目录
    4. .state/redundant_edges：当前，被认为冗余的路径集合
    5. .state/variable_behavior：表现出不同行为的路径
  4. 创建.synced目录，用于跟踪合作模糊测试器的同步目录
  5. 创建crashes目录，该文件夹下保存所有导致crashes的样本
  6. 创建hangs目录，该文件夹下保存所有导致hangs的样本
  7. 创建plot_data文件夹，用于gnuplot输出

maybe_delete_out_dir

该函数只有当创建out目录失败，但是out目录存在的时候被调用，这时候，AFL认为你是想要恢复Fuzz，所以会对out目录做一些操作，根据官方注释：当模糊测试器的输出目录属于我们，并且模糊测试器未运行，且上次运行时间不长的话，就会删除该目录。

  1. 拼接出out_dir/fuzzer_stats
  2. 尝试打开out_dir，如果打不开的话，报错退出，如果能打开，尝试对该目录进行加锁，如果加锁失败，可能被其他进程锁定，报错推出
  3. 打开前面拼接出来的out_dir/fuzzer_stats目录，如果能打开的话，评估一下价值高不高
    这里主要是：看起始时间和最后一次更新时间，看这个时间有没有超过25分钟，如果超过的话报错退出
  4. 如果开了in_place_resume的话，将旧的queue目录移动到_resume，这是恢复的重要一步
  5. 如果没开in_place_resume的话，尝试删除out_dir/.synced，然后报错==退出函数==
  6. 然后，顺序尝试删除/queue/.state/deterministic_done、/queue/.state/auto_extras、/queue/.state/redundant_edges、/queue/.state/variable_behavior、/queue/.state、/queue，如果删除失败，报错==退出函数==
  7. 如果没有开启in_place_resume，删除/crashes/README.txt
  8. 然后就是：重命名备份crashes和hangs文件夹
  9. 最后，尝试删除/.cur_input、/fuzz_bitmap、/fuzzer_stats、/plot_data

read_testcases

该函数主要功能就是：从输入目录中，读取所有测试用例，然后，将它们加入到测试队列，在启动时调用，我们来详细看一下：

  1. 在in_dir下创建queue目录，但是这里，如果已经存在了的话，AFL默认你是想拿整个out当输入（也就是说，AFL会认为，你是将-i -out错误输入成了-i out
  2. 然后如果你设置了随机读取，那么他会首先将数组打乱，否则的话，是严格按照顺序的
  3. 循环读取每一个文件
    1. 为每一个文件都创建一个/.state/deterministic_done/file_name，用来记录该文件是否已经经过确定性变异
    2. 跳过垃圾文件（不是普通文件、空文件、README.testcases）
    3. 超过文件大小限制的也不要
    4. 调用add_to_queue函数（后面分析）

pivot_inputs

pivot_inputs 是 AFL 从“读入初始样本”切换到“自己统治世界”的仪式。
说人话，就是：在该函数之前，input目录可能有很多乱七八糟的目录，也可能是上一次Fuzz的残留，但是经过pivot_inputs函数之后，所有的输入，都将统一，我们来详细看一下该函数实现过程：

  1. 获取队列指针：struct queue_entry* q = queue;，这个queue，我们在前面的read_testcases函数中已经初始化过了
  2. 然后进入循环：
    1. 获取实际的文件名（去除路径）：strrchr(q->fname, '/')，如果失败的话，直接就用q->fname
    2. 匹配文件名是不是id:000000这种格式，且与当前id相匹配（因为都是按照顺序读取的话，id也应该一样的）
      如果是这种格式的话，那么就认为是从上一次的Fuzz恢复的，那么就会：
      <1>. 链接被定向到：out_dir/queue/fname
      <2>. 寻找变异深度（这里对于变异深度，也就是depth，我们到后面看吧，应该看了Fuzz过程之后就能理解了）
    3. 如果不是从上一次Fuzz恢复的，那么就会尝试从文件名中寻找orig:字符串，实际上还是在提取原始文件名，然后拼接出链接路径:out_dir/queue/id;xxxxxx,orig:fname
  3. 做完以上两个分支之后：就会调用link_or_copy函数创建链接（或者copy），==也正是这里，AFL将所有输入都归为自己的资产了==
  4. 然后确保passed_det值也被传递
  5. 最后，删除用于恢复会话的目录

load_extras

当我们在使用afl-fuzz的时候，如果使用-x选项指定了字典，那么该函数就会被调用，用来去加载我们自定义的字典。

  1. 检查文件夹名称中是否带有@，如果有的话，则提取级别并继续
  2. 使用opendir打开文件夹，如果错误为ENOTDIR，则调用load_extras_file尝试以文件加载
  3. 如果打开文件夹成功
    <1>. 拼接完整路径，检查路径有效性
    <2>. 更新min_len和max_len
    <3>. 申请extras内存，直接将文件内容读取进去

load_extras_file

该函数还是为了读取用户自定义的字典，只不过这里比前面要详细一点：

  1. 打开文件，逐行读取文件内容
  2. 去除左右空白，省略行号和#注释
  3. 确保以""开头结尾
  4. 跳过字母数字下划线
  5. 如果是@number，则将number解析为dict_level
  6. 然后就是处理最终过滤出来的字符串，这里就是处理转义和十六进制字节
  7. 写入到extras中

find_timeout

在使用afl-fuzz的时候，如果我们指定了-t选项，那么有可能被设置timeout_given标志位，而如果该标志位==未被设置==，那么就会调用find_timeout函数。
如果这是一次“恢复 fuzz”，那就从上一次的 fuzzer_stats 里，把 exec_timeout 抄回来继续用。

detect_file_args

这里是将参数中的@@替换为当前输入的样本。

@@ 是一个占位符，用于指定目标程序的输入文件位置。当 AFL 执行模糊测试时，它会将测试用例文件动态替换到命令行中的 @@ 位置

setup_stdio_file

为 当前 fuzz 测试用例准备一个临时存储文件。Fuzzer 后续会把测试数据写入 out_fd，然后程序读取它执行目标程序。
我们前面分析的detecet_file_args函数，在替换@@占位符的时候，就会替换成这个文件，用于程序输入。
如果out_file没有值（没有使用-f），则调用此函数，会删除原本的out_dir/.cur_input，创建一个新的out_dir/.cur_input，保存其文件描述符在out_fd中

check_binary

指定路径处要执行的程序是否存在，且它不能是一个shell script，同时检查elf文件头是否合法及程序是否被插桩

get_cur_time

获取当前时间

perform_dry_run(use_argv)

其实到这里已经是AFL的核心代码了。==在阅读该函数之前，先把calibrate_case函数看了，会更好理解。==
首先需要注意的是，该函数传入的参数是：原始参数，去除了附加参数后的其他参数，比如：afl-fuzz -i ./in -o ./out a @@，那么传入的就是：a @@，也就是拿到了要Fuzz的可执行程序的参数，==实际上@@已经被替换为实际参数==。
根据官方注释：对所有测试用例进行预运行，以确认应用程序运行正常。此操作仅针对初始输入执行一次。
那我们就来详细看一下该函数：

  1. 获取环境变量AFL_SKIP_CRASHES：u8* skip_crashes = getenv("AFL_SKIP_CRASHES");
  2. 然后就开始循环读取测试队列
    1. 获取测试用例文件名
    2. 申请内存空间，将测试用例内容读取到该内存中
    3. 调用calibrate_case函数校准测试用例
    4. 如果在校准过程中，stop_soon标志位被设置，直接return
    5. 拿到校验结果(res = calibrate_case(argv, q, use_mem, 0, 1);)
    6. 对校验结果进行分发：
    7. FAULT_NONE：正常运行，无异常，如果是第一个测试用例的话，会调用check_map_coverage函数
    8. FAULT_TMOUT：超时，被认为是校准失败
    9. FAULT_CRASH：如果设置了跳过Crash的话，认为是校验失败，否则检查是否又内存限制，打印详细解释，指导用户调整内存或移除坏样本。
    10. 后面几个直接终止了，这里不一一说了
    11. 指向下一个测试用例
  3. 当测试队列中的样本全部被校准之后，检查是否存在校准失败的样本，如果存在，输出提示

calibrate_case

run_target

根据官方注释：执行目标应用程序，监控超时情况。返回状态信息。被调用程序将更新 trace_bits[]。
也就是说，在调用该函数的时候，会传入一个命令行，这个命令行对应一个样本，而对于这个样本，由于有forkserver的存在，会去记录下bitmap（也就是路径反馈），但是本函数不处理，只负责执行，然后记录下崩溃、超时等信息。
那我们就来详细看一下，该函数具体的做法：

  1. 首先，清空共享内存memset(trace_bits, 0, MAP_SIZE);MEM_BARRIER();  //这里是防止编译器重排，一定是先清零

    设置内存限制等
    设置环境变量
    运行目标程序
    这里需要注意一点，永远回不来

    1. 向管道中写入4字节数据，告诉forkserver上一次是否超时，这时候，forkserver接收到4字节数据，就会fork子进程，开始跑样本
    2. fork之后，会向管道中写入pid，本函数接收保存
    3. 跑完之后，会向管道中写入状态，但是该函数在后面会接收
  4. 然后，本函数设置一个定时器
  5. 如果是没插桩或者没有启用forkserver的话，等待子进程结束，否则，从管道中读取执行状态
  6. 看状态是否有效，无效的话child_pid = 0
  7. 计算运行耗时
  8. 执行次数加一
  9. 然后会从共享内存里读取一个标志，这是一个执行失败标志
  10. 记录本次是否超时，供下次使用
  11. 如果子进程被信号杀死等几个错误return FAULT_TMOUT
  12. 如果超时了的话，slowest_exec_ms = exec_ms;
  13. 正常执行完之后，return FAULT_NONE;

call_queue

根据官方描述：裁剪队列上述机制的第二部分是一个例程，它会遍历 top_rated[] 条目，然后依次抓取之前未见过的字节（temp_v）的优胜者，并将它们标记为“优先”，至少在下次运行之前如此。这些优先条目在所有模糊测试步骤中都会获得更多的运行时间。
完全不理解，我第一次看的时候，这是啥玩意啊。
首先，我们来看个数据结构，就是top_rated：

可以看到，top_rated这个玩意，实际上就是一个超级大的，样本数组（注意是数组），而这个数组有多大呢？位图中的每一个字节，就有一个queue_entry，可能说到这里，大家还是不太理解，它的功能是：在为图中的每一个字节（这里代表的是路径），都存在一个样本，这里记录的是：这个路径上的最优样本（至于怎么评判最优，AFL有自己的评判标准，我们后面看）。
也就是说，位图是一个地图，在这个地图上我们能看到所有的地点，而top_rated，就是：能够到达这个地点的最优选择（虽然这个最优选择的目的可能不是该地点）
那我们现在来仔细看看，该函数做了哪些操作：

  1. 申请temp_v，这里大小是[MAP_SIZE >> 3]，为什么呢？因为一个字节是8bit，一个bit代表了一条路径
  2. 如果是dumb_mode或者!score_changed，直接return
  3. 然后遍历top_rated数组，将其中favored字段全部清零
  4. 开始循环读取这些（高质量样本）
    1. 如果这条路径没有被别人认领也就是在temp_v中没有标记的话
    2. 进入循环，讲该样本覆盖到的路径，在temp_v(地图)中全部标记出来
    3. 然后将该样本的favored标志位进行标记
    4. 队列计数器加一
    5. 如果这个关键样本还没被Fuzz过，那么pending_favored++
  5. 循环结束之后，将数组中的所有样本的favored标志位进行标记
  6. 那为什么说这个函数是裁剪队列呢？就是因为mark_as_redundant函数的存在

mark_as_redundant

该函数功能较为简单：就是对队列中，无意义（或者说价值不高）的样本进行从链表上删除

show_init_stats

show_init_stats函数是AFL在初始化阶段（完成初始测试用例加载和校准后）调用的一个函数，主要功能是：

  • 统计当前队列中所有测试用例的基本指标
  • 根据这些指标调整fuzzing策略参数
  • 向用户输出警告或建议，以优化fuzzing效率
    那我们就来详细看看，该函数具体是怎么做的：
  1. 计算执行周期平均消耗时间
  2. 进入循环，统计：所有样本中，最大、最小运行时间、样本长度
  3. 检查平均耗费时间是不是过长
  4. 根据平均耗时，设置havoc_div标志位
  5. 根据是否dumb模式，给用户一些建议
  6. timeout设置，这里是：慢程序 → 系数小、快程序 → 系数大

find_start_position

trim_case

为了节省确定性检查的计算周期，所有新增的测试用例都会进行精简。 精简器使用文件大小的 1/16 到 1/1024 之间的 2 的幂次方作为增量，以确保该阶段快速高效。
这个函数就是裁剪样本的函数，确定从样本中删除哪些东西，不会影响走过的路径，以提高样本变异效率，裁剪逻辑：首先选取一个步长，窗口滑动进行删除，根据位图获取hash，如果路径没变，就说明该段是可以不存在的，就删除，以此类推，每次步长减小。

calculate_score

该函数就是对样本进行一个评分，在整个样本队列中，进行评分，基于时间、代码覆盖率和执行深度三种评分标准，当然，也不是一棒子打死，哪些评分靠后的样本，也有机会超车。

save_if_interesting

该函数接收4个参数：argv、当前程序读取内存、长度和执行返回结果。
根据返回结果，如果是crash_mode（这正是我们感兴趣的），那么就将当前样本加入到测试队列，并写入文件
超时并不是很有趣，但我们仍然有义务保留少量样本。如果没有发生新的路径，那么return keeping，但是如果发现新的路径，且运行时间超过hang_tmout，那么将保存到hangs目录下
如果返回结果是FAULT_CRASH：那么不需要重新运行测试用例，写入文件（crashes目录下）

Fuzz_one

队列中取出当前条目，对其进行一段时间的模糊测试。此函数略显冗长……模糊测试成功则返回 0，跳过或中止则返回 1。
这个函数代码量比较大，这里就不粘贴代码了，直接来讲述一下该函数做的事情：

  1. 看看队列中有没有要优先考虑的，如果有，经过两个裁剪队列逻辑（注意不是裁剪样本逻辑），有概率跳过当前样本
  2. 读取当前样本到orig_in和in_buf，读取当前样本深度
  3. 如果当前样本校准失败（根据之前的queue_cur->cal_failed标记），尝试再次校准
  4. 如果不是恢复模式，且该样本没有被修剪过，那么就调用trim_case函数进行修剪（注意这里，只能修建一次，即使失败，也不再修剪）
  5. 然后memcpy(out_buf, in_buf, len);，现在这里的in_buf，有可能是被修剪过的，也有可能还是原样
  6. 通过calculate_score函数获取当前样本的评分
  7. 如果用户指定跳过确定性变异，或者当前样本已经执行过，或者说之前已经通过确定性变异阶段，直接跳转到随机变异部分，  如果执行路径校验和超出此主实例的范围，则跳过确定性模糊测试
  8. 设置doing_det = 1;，标记开始确定性变异
  9. 单bit翻转，遍历文件的每一个bit并进行翻转，然后运行程序（common_fuzz_stuff），看是否有新发现
  10. 双bit翻转、四bit翻转和8bit翻转，==这里8bit翻转有个不同的点，会被认为：即使字完全翻转也不会影响任何字节，那么在确定性变异阶段就会跳过他们（这里是通过eff_map完成的）==，后面还有双字翻转，四字翻转，8字翻转，
  11. 接下来进行算数变异（单字节，双字节，4字节），如果eff_map标记为0，就会跳过，如果算术运算的结果可能已经被位翻转达到了，也会跳过，对样本每一个字节进行：+1,-1,+2,-2……+34,-34,+35,-35
  12. 有趣值替换，对样本每一个字节（双子，四字，八字）循环替换为内置的有趣值（包括临界值等等）
  13. 随机变异，轮数叠加，随机：翻转，裁剪，字典替换等等
  14. 拼接 splicing：havoc 没什么效果就会进入 splicing，拼接两个测试用例再重复上述变异

巧妙算法设计

1. FLIP_BIT

该宏被定义于fuzz_one函数该宏是用于字节翻转，我们先来看一下这个代码：

阅读fuzz_one代码，我们可以知道，传入的是样本内存的指针（_ar永远是头指针，_b是偏移量，注意是bit偏移，而不是字节偏移）。
前两句转换我们就不解释了，很简单，我们主要看最后一句：_arf[(_bf) >> 3] ^= (128 >> ((_bf) & 7));
首先，_arf[(_bf)>>3]，_arf是样本的头指针，这里的(_bf) >> 3，由于我们传入的_b是bit的偏移量，所以对其除以8，就得到了字节的偏移量，这样，我们就得到了要反转的某个bit，在样本头开始的多少字节，这样，我们确定了一个字节，我们要反转的bit，就在这个字节的8bit中，接下来，我们来看看是怎么在这8个bit中查找目标bit的：
这里(_bf)&7，这个其实是寻找在特定byte中特定bit的算法（这里其实是取模运算的位写法）
其次，利用128 >> ((_bf)&7)，128的二进制：1 0 0 0 0 0 0 0，正好是8位，由于我们知道我们要翻转的bit在这8bit中的第几位，所以我们利用128 >> ((_bf)&7)，就能将1 0 0 0 0 0 0 0中的1，放到期望的位置
最后，利用^=算法，完成翻转。

现在我们来梳理一下，在循环中，锁了什么事情：

Fuzz流程梳理

那我们现在来梳理一下main函数做的事情，这样，我们可以清楚地知道，Fuzz过程中做的事情：

  1. 解析命令行参数，针对不同的参数，设置不同的标志位
  2. setup_signal_handlers：注册信号处理函数，Fuzzer有能力感知子进程的退出状态（主要通过触发信号时，设置标记位来实现）
  3. check_asan_opts：检查ASAN选项，当检测到内存错误时，直接退出，这里还是主要是为了提高Fuzzer效率，同时让Fuzzer有能力感知这些错误
  4. fix_up_sync：以便多个 Fuzzer 实例可以在不同的机器或核心上并行工作，并共享测试成果。
  5. save_cmdline：保存原始命令行参数
  6. fix_up_banner：修剪，以便在控制台显示
  7. check_if_tty：检查是否运行在tty（以决定是否在控制台显示）
  8. get_core_count：获取CPU核心数等
  9. check_crash_handling：确保核心转储不会被转发到程序
  10. check_cpu_governor：检查CPU频率调解器
  11. setup_post：加载postprocessor
  12. setup_shm：创建共享内存
  13. init_count_class16：简化路径信息反馈处理，具体的可以看前面的文章
  14. setup_dirs_fds：创建所需要的目录，其中：如果设置了sync_id，需要创建同步目录，之后，创建输出目录，并对输出路进行加锁、创建.../queue目录，保存起始路径和已经发现路径的队列目录、创建.../queue/.state/目录，用来保存状态，也用于恢复模式、创建.../queue/.state/deterministic_done/目录，用于标记过去经过确定性模糊测试的队列条目目录、.../queue/.state/redundant_edges/目录，当前被认为冗余的路径集合、创建.../queue/.state/variable_behavior/目录，用来保存表现出不同行为的路径
  15. read_testcases：从输入目录读取文件到输入队列中（queue_top)
  16. load_auto：加载自动生成的额外内容
  17. pivot_inputs：在输出目录中为输入测试用例创建硬链接（可以好好看看源码，这里就是起个别名，但是对AFL还是很重要的）
  18. load_extras：加载用户字典（如果有的话）
  19. detect_file_args：替换命令行中的@@字符
  20. setup_stdio_file：这里实际上是双重路径的第二个，如果没有@@，那么就会尝试去从.../.cur_input去读取
  21. check_binary：这里校验可执行程序是一个程序而非脚本
  22. get_qemu_argv：为qemu模式重写argv，实际上就是替换到qemu模式的Fuzzer
  23. perform_dry_run：对输入目录进行校准
  24. find_start_position：如果是恢复模式，找到上一次运行的位置
  25. Fuzz主循环（这个就在上面，直接翻上去看吧，这里不写第二遍了
  26. 写入位图，更新统计文件
